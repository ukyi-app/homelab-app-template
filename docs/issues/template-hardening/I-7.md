# I-7 — template-ci 강화: 스모크 런·SHA 핀·bun 버전 정합

## Parent

`docs/prds/template-hardening.md`

## What to build

템플릿 드리프트 가드가 "빌드는 통과하나 기동은 실패"하는 회귀를 잡게 만든다.
현재 template-ci는 4개 아키타입 이미지를 build만 하고 **한 번도 기동하지 않아**,
빌더/런타임 베이스의 glibc 갭이나 엔트리포인트·경로·포트 어긋남 같은 런타임 파손이
CI green으로 통과한다.

- **스모크 런(4개 아키타입 전부)**: 빌드된 이미지를 실제로 기동해 런타임 계약을
  검증한다.
  - web/api: liveness·readiness 경로가 200을 반환한다(api는 무DB 시 readiness가
    정적 통과하므로 부가 인프라가 필요 없다).
  - worker: 기동 로그 확인 + **sleep 도중 SIGTERM을 보내** 제한 시간 내(수 초)
    종료·종료 코드 0·종료 로그를 단언한다(I-1의 중단 가능 sleep 직접 검증).
  - site: 최소 인자(포트·루트·헬스)만 주입해 헬스 경로를 확인한다 — 차트의 세부
    인자 재현은 하지 않는다.
- **액션 SHA 핀**: template-ci가 서드파티 GitHub 액션을 이동 태그로 참조한다
  (homelab은 전부 커밋 SHA 핀). 이 워크플로는 포크 PR에서도 실행되므로 태그 이동/
  탈취가 공급망 표면이 된다. 커밋 SHA(+버전 주석)로 핀한다.
- **bun 버전 정합 grep 가드**: bun 버전이 4개 Dockerfile과 CI 설정 등 여러 곳에
  하드코딩되어 있어, 부분 범프가 "lock 재생성 bun ≠ frozen-lockfile 빌드 bun"을
  조용히 만들 수 있다. 모든 지점의 버전이 일치하는지 검증하는 스텝을 추가한다
  (불일치 시 실패).
- **DX 존재 계약**: 스캐폴드된 fullstack에 dev 스크립트와 concurrently
  devDependency가 존재함을 단언한다(기존 계약-검증 스타일). `bun run dev`를 실제로
  기동하는 프로세스 seam은 두지 않는다(PRD P-5 결정 — 플레이크 위험).

## Acceptance criteria

- [ ] 4개 아키타입 각각에 대해 빌드된 이미지를 기동하는 스모크 스텝이 있고, CI가
      green이다.
- [ ] web/api 스모크가 liveness·readiness 200을 확인한다.
- [ ] worker 스모크가 sleep 도중 SIGTERM에 대해 제한 시간 내 종료·코드 0·종료
      로그를 단언한다(I-1 이전 코드였다면 실패했을 검증).
- [ ] site 스모크가 최소 인자로 기동해 헬스 경로를 확인한다.
- [ ] template-ci의 서드파티 액션이 모두 커밋 SHA로 핀되어 있다.
- [ ] bun 버전이 어느 한 곳이라도 불일치하면 CI가 실패한다(의도적 불일치로 확인).
- [ ] 스캐폴드된 fullstack에 dev 스크립트·concurrently devDep이 있음을 단언한다.
- [ ] 기존 검증(계약 스키마, 네거티브, 풀러-안전 가드, 타입검사)이 모두 유지된다.

## Blocked by

- I-1 (worker SIGTERM 종료 동작이 존재해야 스모크가 통과)
- I-2 (Dockerfile 게이트·동봉 테스트가 존재해야 함)
- I-5 (fullstack dev 스크립트·concurrently가 존재해야 단언 가능)
