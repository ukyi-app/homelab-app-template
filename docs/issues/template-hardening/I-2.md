# I-2 — app 팩토리 + 엔트리 가드 + 동봉 테스트 + Dockerfile 게이트

## Parent

`docs/prds/template-hardening.md`

## What to build

스캐폴드된 앱이 **처음부터 테스트 가능한 구조**로 출발하고, 이미지 빌드 자체가
품질 관문이 되게 한다(ADR-0001).

- **테스트 seam(유일한 신규 seam)**: api·fullstack의 서버 모듈이 Hono app을
  export하고, 서버 기동은 엔트리 가드(`import.meta.main`) 안으로 옮긴다 — import
  부수효과 없이 `app.fetch`로 라우트를 검증할 수 있게 된다.
- **api는 팩토리**: app 생성이 DB 풀을 주입받는다(기본값은 env에서 자동 발견된 풀).
  이 주입 지점이 I-3의 DB 실패·복구 검증을 가능하게 하는 seam이다. 단, 리스너
  등록 등 풀 준비 로직의 구체 구현은 I-3의 몫 — 여기서는 주입 가능한 형태만 만든다.
- **동봉 테스트**: api·fullstack에 liveness·readiness·예제 API 라우트 스모크
  테스트(app.fetch), api에 DB URL 자동 발견 로직(접미사 매칭, MIGRATE_/RO_ 제외,
  generic fallback)의 단위 테스트. worker·site는 테스트 표면이 없어 동봉하지 않는다.
- **Dockerfile 게이트**: 4개 아키타입 모두 빌드 앞에 typecheck를, 테스트를 동봉한
  api·fullstack은 테스트 실행을 추가한다. 타입 에러나 테스트 실패는 이미지 빌드를
  실패시켜야 한다 — release 워크플로는 thin-caller로 유지하고 preflight 잡은
  만들지 않는다.

## Acceptance criteria

- [ ] api·fullstack의 app을 import해도 서버가 기동되지 않고, `app.fetch`로
      liveness·readiness·예제 API 라우트를 검증할 수 있다.
- [ ] api의 app 생성이 DB 풀을 주입받을 수 있다(기본값: env 자동 발견).
- [ ] `bun test`가 api·fullstack에서 통과한다(라우트 스모크 + api의 URL 자동 발견
      단위 테스트).
- [ ] 4개 아키타입 Dockerfile 빌드가 typecheck를 통과해야만 성공한다 — 의도적으로
      타입 에러를 넣으면 빌드가 실패함이 확인된다.
- [ ] api·fullstack Dockerfile 빌드가 테스트 실패 시 실패한다.
- [ ] 기존 런타임 계약이 보존된다: 직접 실행 시 여전히 :8080에서 서빙하고
      metrics는 :9090에 뜬다. I-1의 종료 동작이 깨지지 않는다.
