# I-3 — pg 풀 오류 격리 + DB 복구 시퀀스 검증

## Parent

`docs/prds/template-hardening.md`

## What to build

api 아키타입에서 DB 순단(풀러 재시작, cnpg 페일오버, 네트워크 파티션)이 프로세스
크래시가 아니라 **readiness 503 → 자동 회복**으로 표출되게 한다. 현재는 pg 풀에
error 리스너가 없어 유휴 커넥션 오류가 unhandled 이벤트로 프로세스를 즉사시키며,
이는 "liveness는 정적 — DB 일시장애로 파드가 죽지 않게"라는 코드 자신의 설계
의도를 정면으로 위반한다.

- **공유 풀-준비 경로**: error 리스너 등록은 풀 준비 함수 **한 곳**에만 존재하고,
  팩토리가 env에서 만든 풀이든 테스트가 주입한 풀이든 **같은 준비 경로**를 통과한다
  (PRD P-6 결정 — 주입이 프로덕션 리스너 등록을 우회하지 못하게).
- 리스너는 유휴 커넥션 오류를 로깅만 하고 삼킨다. 순단은 다음 readiness 왕복에서
  503/재연결로 자연 수렴한다.

## Acceptance criteria

- [ ] 풀 준비 경로가 하나뿐이고, 프로덕션 생성 풀과 테스트 주입 풀이 모두 그 경로를
      통과해 error 리스너를 얻는다.
- [ ] **복구 시퀀스 테스트**(상태 있는 스텁 풀 하나 + 단일 app 인스턴스로 전 구간):
      풀이 error 이벤트를 emit해도 프로세스가 생존한다 → 그 상태에서 readiness가
      503을 반환한다 → **같은 풀**이 성공 상태로 전환되면 readiness가 200을 반환한다.
- [ ] 무DB(풀 없음) 분기의 readiness 200이 별도 케이스로 검증된다.
- [ ] liveness는 DB 상태와 무관하게 항상 200이다.
- [ ] 임시 Postgres/PgBouncer를 띄우는 이미지 레벨 DB 테스트는 도입하지 않는다.
- [ ] `bun test`와 Dockerfile 게이트가 통과한다.
