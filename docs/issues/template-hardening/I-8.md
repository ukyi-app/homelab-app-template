# I-8 — 템플릿 전용 문서가 생성 앱으로 새어 나가지 않게 한다

## Parent

`docs/prds/template-hardening.md`

## What to build

앱 레포는 이 템플릿을 통째로 복사해 만들어지므로, 템플릿 레포에 커밋된 모든 파일이 앱 레포의
초기 내용이 된다. 이 배치는 `CONTEXT.md`(템플릿의 용어집)와 `docs/`(PRD·이슈·리뷰 아티팩트·ADR,
15개 파일)를 추가했다 — 즉 **이 파이프라인 자신의 서류가 앞으로 생성되는 모든 앱 레포에
복사된다**. 앱 개발자에게는 남의 레포 회의록이고, 앱의 커밋 히스토리·검색·에이전트 컨텍스트를
오염시킨다.

스캐폴더는 이미 템플릿 전용 머신러리를 자가삭제한다(`scaffold/`, `.github/workflows/template-ci.yaml`,
그리고 I-7이 추가한 `.bun-version`). 템플릿 전용 문서도 같은 범주다.

- 무엇이 템플릿 전용인가: 파이프라인 아티팩트(`docs/prds/`, `docs/issues/`, `docs/reviews/`)는
  명백히 그렇다. `docs/adr/`와 `CONTEXT.md`는 **템플릿의** 결정·용어를 담는다(앱의 것이 아니다).
- 무엇이 앱의 것인가: 앱 README(`README.app.md`에서 렌더), `.env.example`, `renovate.json`,
  `.github/workflows/release.yaml` — 이미 `scaffold/common/`에서 온다.
- 판단이 필요한 지점: ADR-0001(Dockerfile 게이트)·ADR-0002(앱 식별자)는 앱 개발자가 "왜 이렇게
  되어 있나"를 물을 때 유용할 수 있다. 통째로 지울지, 앱에 필요한 내용만 앱 README가 이미 설명하는지
  확인하고 결정하라.

**주의**: 삭제 지점은 `bun install`(lock 재생성) **이후**여야 한다. 그 앞에 두면 실패 롤백이
복구하지 못하는 '지워진 원본'이 생겨, I-4가 정직화한 롤백 주석·실패 메시지가 다시 거짓이 된다
(scaffold.ts의 자가삭제 블록에 이 제약이 이미 주석으로 못박혀 있다 — 그 자리에 함께 둘 것).

## Acceptance criteria

- [ ] 4개 아키타입 어느 것으로 스캐폴드해도 생성된 앱에 `docs/prds/`·`docs/issues/`·`docs/reviews/`가
      존재하지 않는다.
- [ ] `CONTEXT.md`와 `docs/adr/`의 처리 방침이 결정되고 그 근거가 코드 주석 또는 이슈 결과에 기록된다
      (지우든 남기든, 이유가 있어야 한다).
- [ ] 앱에 필요한 파일은 그대로 남는다: README.md, .env.example, renovate.json,
      .github/workflows/release.yaml, .app-config.yml, tsconfig.json, .gitignore, .dockerignore,
      bun.lock, package.json, 앱 소스.
- [ ] 템플릿 레포 자신은 온전하다 — `docs/`와 `CONTEXT.md`는 템플릿에 남고, template-ci와
      I-5의 드리프트 가드가 계속 동작한다.
- [ ] 삭제가 `bun install` 성공 이후에 일어나며, 실패 롤백 시나리오에서 I-4의 롤백 주석·실패 메시지가
      여전히 사실이다(강제 실패로 확인).
- [ ] 4개 아키타입 스캐폴드·docker build·동봉 테스트·네거티브 테스트가 모두 유지된다.
